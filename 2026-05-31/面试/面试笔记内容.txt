Q：JVM中有哪些垃圾回收算法
A：1、标记-清除算法，指的是会将JVM中的可回收对象就行标记，全部标记完成后，再一次性的全部清除。但要注意这种算法会产生很多的内存碎片，
当我们大对象需要分配一块连续的内存空间时，就会有问题，再次触发GC。
2、复制算法：它会将内存空间划分为A和B的2块，A内存空间使用，B内存空间暂不使用，当A块发生GC时，就会把空间中存活的对象复制到B内存空间中，
然后一次性清除A内存空间。这种算法的内存空间使用率其实是不高，因为总有一块内存空间时没有在使用的，效率是想对好点
3、标记-整理算法：和第一种算法一样会对回收对象进行标记，回收对象之后，会对内存空间进行整理，这样内存空间中就会有连续的内存空间了。

 4种隔离级别分别是：读已提交、读未提交、可重复读、串行化。
 1、读未提交是A事务会读取到B事务未提交的修改数据，这就导致出现脏读。

 注意前面3种隔离级别都会出现幻读的情况，幻读是事务A前后读取表的总条数，会发生变化。
 4、串行可以解决幻读的情况，这个隔离级别是最高的，它要求事务的操作必须串行化执行，只有上个事务执行完成之后，下一个事务才能执行。

线程安全需要满足哪些条件：可见性、原子性、有序性

1、原子性中有最典型的I++操作，就不是原子性，
i++操作可以拆分成3个步骤，读取I的值，给I的值加1，将最新的值赋值给i。
如果在并发过程中，i的值被其他线程给赋值的话，那最终输出的结果就不对了
2、可见性问题指的是：当一个线程修改了共享变量的值，其他线程无法立即看到这个修改。
例子的就有while（flag）,主线程更新变量值，但是子线程有可能一直获取不到，导致卡死。
主要原因是在Java的内存模型中，会分为主内存和工作内存，每个线程中有自己的工作内存，每个线程中的变量值都是从主内存中获取的一个副本，
并且每个线程只能在自己的工作内存中操作变量，是不会在主内存中操作变量的。这就导致了最新的值只有自己的线程才知道，我们可以使用volatile
关键字来修饰变量,volatile会将我们的工作内存中的刷新到主内存中，同时将其他线程中变量值失效，这样其他线程需要使用变量的时候就从
主内存中获取最新的。这也是volatile保证可见性的原理，同时volatile也能禁止指令重排序，是通过内存屏障完成的。需要注意的是volatile
不能保证原子性
3、有序性：指的JMM中会存在优化重排序的操作，将我们对象的操作顺序发生变化。


线程安全的五种实现策略

├── 1. 不共享：单线程中使用变量
├── 2. 不可变：我们可以通过final修饰变量，
├── 3. 线程封闭：在多线程中我们可以使用线程私有对象，ThreadLocal创建对象，ThreadLocal能确保变量值只会在一个线程中。
├── 4. 同步：使用Synchronization和ReentrantLock加锁的方式保证线程安全
└── 5. 并发容器 / 原子类：使用AtomicInteger或ConcurrentHashMap这种线程安全的类


AtomicInteger实现线程安全主要是通过CAS来保证的，CAS它是一种并发的算法，逻辑是包含三个操作数，读写需要的内存位置A，进行比较的值B和最新的值C
只有当进行比较的值B等于内存位置A的，才会使用最新的值C去更新A。
不过这种CAS算法会出现ABA的现象，虽然变量值前后是不变的，但是变量值中途可能会发生改变。基于这种情况我们通常使用的方式加一个版本变量，B的值每次发生变化，版本加一，在原本的基础上新增判断版本的值是否有发生变化

MVCC原理
MVCC的全称是多版本并发控制，在MVCC中维护了版本链的概念，
要想理解版本链就需要先了解在我们的主键索引列中有2个隐藏列
一个是事务ID列，一个是指针列。每次我们对记录进行修改的时候，
都会把旧的版本写入到Undo log（回滚日志）中，指针列就会存放指向我们
旧的版本信息。随着我们更新的版本次数增多，所有的版本都会被指针列连接成一个链表，这就是版本链。

如果是隔离级别是读未提交的话，就直接从版本链中获取最新的值就行了。
如果是读已提交和可重复读的话，就需要用到readVIew（一致性视图，当前快照版本的视图）。
readVIew能帮我们判断读那个版本的数据，它主要有4个内容
1、m_ids ：当前系统中活跃的事务ID列表，其实就是未提交的事务ID列表
2、min_trx_id：活跃事务列表中最小的事务ID
3、max_trx_id：系统应该分配给下一个事务ID的值
4、creator_trx_id：生成该事务的ID值。

MySQL只有在执行到增删改的时候事务才会分配ID，查询的事务ID值为0

readVIew判断逻辑如下：
1、如果被访问版本的trx_id属性值与ReadView中的create_trx_id值相同，说明是当前事务自己访问自己的数据，可以访问
2、如果被访问版本的trx_id属性值小于Readview中的min_trx_id，则表明生成这个版本的事务ID已经提交了，可以访问
3、如果被访问版本的trx_id属性值大于或者等于Readview中的max_trx_id，则表明生成这个版本的事务ID在readview之后生成，不可以访问
4、如果被访问版本的trx_id属性值在Readview中的min_trx_id和max_trx_id之间，就判断是否在m_ids列表中，如果在，说明生成这个版本的事务ID
还未提交不可以訪問，如果不在则说明事务ID已经提交了，可以访问


读已提交是每次在执行到select的时候都会生成一个readview,如果读已提交的事务有多个查询，那么就会触发多个readview。
而可重复读只有在第一次执行select的时候生成唯一一个readview。

为什么MVCC无法解决幻读


为什么“当前读”是解决幻读问题的关键？
回到我们之前的讨论，纯MVCC无法解决幻读，正是因为在同一个“可重复读”事务中，存在从“快照读”到“当前读”的模式切换。
让我们再简化一下那个经典例子：
事务A（RR级别）：SELECT * FROM t WHERE id > 100; // 快照读，假设返回空集。
事务B：插入 id=150 并提交。
事务A：UPDATE t SET name=‘a’ WHERE id > 100; // 执行此语句时触发当前读！
为了执行更新，引擎必须找到“此刻”数据库中所有 id > 100 的行。
它看到了事务B提交的 id=150 这一行，于是将其作为更新目标。
此时，id=150 这条记录被事务A修改了。根据MVCC规则，一个事务总能看见自己做的修改。
事务A：SELECT * FROM t WHERE id > 100; // 再次查询，此时就能看到自己修改过的 id=150 了。
结论：幻读的“幽灵”（id=150）正是通过 UPDATE 语句中的当前读，被“召唤”进了事务A的视野。如果整个事务只有快照读，这个幽灵是永远不会出现的。
所以，数据库如何堵上这个漏洞？
以MySQL InnoDB为例，它在“可重复读”级别下，对“当前读”操作附加了强大的锁——间隙锁/Next-Key Lock。
当事务A执行 UPDATE t SET ... WHERE id > 100 时，InnoDB在进行“当前读”的过程中，不仅会锁住未来可能读到的行，还会锁住 id > 100 这个整个范围（间隙）。这样一来，事务B的插入操作在步骤2就会被阻塞，根本提交不了。幻读在根源上就被杜绝了。


MVCC能否解决幻读？
分2种情况：
1、快照读：如果整个事务中只是简单查询某个字段值的话，查询的都是在MVCC中版本链上某个版本的数据，这种情况是可以解决幻读的。

2、当前读：假设2个事务中，事务A的流程是先查询，更新或删除再查询。事务B在事务A更新或删除操作之前新增了一条数据，并且这个数据是符合
事务A更新或删除的条件，这种情况下事务A会操作到事务B的新增数据。这样就事务A就会出现幻读。MVCC是无法解决幻读的，通常做法是对
数据行加锁，例如sql语句：select XX for update 


MySQL的行级锁：记录锁或者间隙锁，就看自己的查询的where条件是什么了
使用 select  for update  能够对MySQL进行加锁 ，其他事务ID的会阻塞，这样也能解决幻读

让我详细解释这个问题和解决方案：

1、防腐层和Adpert层

布隆过滤器特点
1、效率高，本质上是一个bitmap，不存储原本的数据，存的都是通过对key值Hash计算之后的0或1
2、存在假阳性，如果布隆过滤器说存在，那有可能是误判。不存在假阴性，如果布隆过滤器说不存在，那么就一定不存在。


set 或list 使用contain方法判断是否存在过，但是如果你存的数据一下子变多了，那你对内存就会有很大的压力。
你可能会想到模仿hashMap那样使用hash计算key值然后将结果保存到一个数组中，这个方法虽然能够减少内存的压力，
但是会出现我们说的hash冲突。要想解决这个问题，你可能会想到一次性使用多个hash函数，得到的多个hash结果保存在数组中

使用hash函数得到整数结果后，我们就把这个整数在数组中的位置为1，表示存在

布隆过滤器本身提供一个二进制向量，初始值都是0。hash结果得到一个整数后，就将这个整数位置的二进制向量值改为1，
如果本来就为1就不用修改。所以1表示存在，0表示不存在，因为会存在hash冲突，单纯靠一个位置的值判断是不准确的，
为了减少这种情况我们可以一次使用多个hash函数，就会映射多个位置，多个位置判断能够减少我们误判的可能。


黑名单过滤，用户重复请求，推荐物品的曝光过滤，避免缓存穿透（就是快速过滤用户输入哪些不存在的Redis的key值）

布隆过滤器缺点： 整个二进制向量是共享的，如果你要移除某个key的1值，会影响到其他key判断



第一部分：DDD是什么（概念解释）
面试官您好，我对DDD有一定了解。DDD（领域驱动设计）是一种软件设计方法论，核心思想是：

聚焦业务领域：以业务需求为核心，而不是技术实现

统一语言：开发团队和业务专家使用共同的术语沟通

模型驱动：将业务概念转化为软件模型，模型反映业务本质

分层架构：清晰划分职责，保持领域逻辑的纯粹性

我理解的DDD包含两个层面：

战略设计：划分业务边界（限界上下文）、识别核心子域、通用子域等

战术设计：在限界上下文内使用实体、值对象、聚合、领域服务等模式

在1.7中使用分段锁来保证线程安全，本质上锁的还是一个子数组，内存开销较大。所以在1.8中做了优化
在1.8中使用CAS加synchronized 来保证线程安全，如果我们数组中槽位是空的，使用CAS无锁操作
效率更快，如果槽位上有值，使用synchronized加锁，synchronized锁的是链表上的头节点或者树的根节点，使用synchronized主要是因为在 JDK 1.8 之后做了大量优化（如锁升级），
从偏向锁-》轻量级锁-》重量级锁，重量级锁一般情况下是使用不到重量级锁的。使用偏向锁和轻量级锁效率也是很快的
